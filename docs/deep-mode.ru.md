# Deep-режим (атомизация + проверка)

У Mnemazine два режима работы:

- **Консервативный (по умолчанию):** только локально. Без сети, без LLM, без внешних сервисов. Так работают `node scripts/mnemazine-run.mjs` и `npm run synthesize` по умолчанию.
- **Deep (по запросу):** использует локального агента Codex, чтобы разбить один источник на много сфокусированных заметок (README «один источник → ~20 заметок») и проверить утверждения по их источникам.

Deep-режим **выключен, пока вы его не попросите**. Ничто в стандартном конвейере не выходит в сеть и не зовёт LLM.

## Включение deep-режима

```bash
# весь прогон, deep:
node scripts/mnemazine-run.mjs --deep
# или через env (форвардится в synthesize):
MNEMAZINE_DEEP=1 node scripts/mnemazine-run.mjs

# только синтез, deep:
npm run synthesize -- --deep
```

Если deep-режим запрошен, но Codex недоступен, прогон **откатывается на локальный шаблонный синтез** и сообщает `degraded: true` в JSON-выводе. Молча не падает.

## Мост Codex

Все вызовы LLM идут через один провайдер-абстрактный модуль: `scripts/mnemazine-llm.mjs` (`llmJson(prompt, schema, {provider, tools})`). **Движок в первую очередь — Claude** (headless `claude -p`); **Codex держится в паритете** — тот же контракт, поэтому всё, что работает через Claude, работает и через Codex. Третьего LLM-клиента нет. `mnemazine-codex.mjs` остаётся тонким shim'ом обратной совместимости.

### Переменные окружения

| Переменная | По умолчанию | Назначение |
|----------|---------|---------|
| `MNEMAZINE_LLM` | `claude` | Движок: `claude` (primary) или `codex` (паритет). |
| `MNEMAZINE_CLAUDE_BIN` | `claude` (PATH) | Путь к Claude CLI. Зафиксируйте для headless-прогонов. |
| `MNEMAZINE_CODEX_BIN` | `/Applications/Codex.app/Contents/Resources/codex` | Путь к бинарю Codex. |
| `MNEMAZINE_LLM_TIMEOUT_MS` | `420000` | Таймаут одного вызова. |
| `MNEMAZINE_DEEP` | не задано | `1` включает deep-режим (enrich + атомизация + проверка + digest). |
| `MNEMAZINE_ENRICH` | `1` в deep | `0` (или `--no-enrich`) пропускает стадию обогащения. |
| `MNEMAZINE_MAX_ATOMS` | `20` | Лимит атомов на один кластер источника. |
| `MNEMAZINE_OWNER_CONTEXT` | общий | Контекст личных проектов для блока «почему важно» в справке. Либо положите в gitignored `.mnemazine/owner-context.txt`. |

### Обогащение (расширение знаний)

Перед атомизацией deep-режим запускает web-агента LLM, который **исследует и расширяет** захваченный материал «насколько реально полезно» — первоисточники, актуальные факты/версии, опыт практиков — каждый добавленный факт привязан к fetched-URL (антигаллюцинация). Атомы строятся из **расширенного** знания, не только из сырого захвата. Отключение: `--no-enrich` / `MNEMAZINE_ENRICH=0`.

### Справка (русская человеческая сводка)

После Graphify `scripts/mnemazine-digest.mjs` (`npm run digest`) пишет в каждую заметку русскую **Справку** в стиле humanizer — *Что это / О чём / Почему важно мне / Связи* — с реальными связями из графа, плюс одну ноту-сводку прогона с картой всех атомов. Это поверхность переиспользования: открыл одну заметку — понял знание и как оно связано. Идемпотентно (пропускает заметки со Справкой, если не `--force`).

### Атомизация (G4)

`scripts/mnemazine-synthesize.mjs` (с `--deep`/`--atomize`) отправляет каждый кластер источника в Codex и просит сфокусированные атомы — у каждого заголовок, что/зачем, пункты «как применить», следующее действие и поддерживающие URL источников. Каждый атом становится отдельной заметкой. Имена файлов — content-fingerprint (с привязкой к id кластера), поэтому повторные прогоны идемпотентны и не затирают существующие заметки.

### Проверка (G5)

`scripts/mnemazine-verify.mjs` присваивает каждой заметке `verification_status`:

- `unknown` — нет URL источника, подтверждающего утверждение;
- `assumed` — URL источника есть, но не запрашивался/не проверялся (**локальный** вердикт по умолчанию, без сети);
- `verified` — только при `--deep`: источник был достижим (HEAD/GET) **и** web-сверка Codex признала, что он подтверждает утверждение. Такие заметки получают `verified: true` и `status: final`.

## Безопасность

### Недоверенный ввод изолируется

Извлечённый материал (OCR, транскрипты, текст с сайтов) — **недоверенный**. Перед попаданием в любой промпт Codex он оборачивается `fenceUntrusted()` — делимитер инертных данных плюс явная инструкция, что содержимое нельзя исполнять как команды. Любое буквальное вхождение sentinel-метки внутри контента нейтрализуется. Это основная защита от prompt-injection через захваченный материал.

### Песочница

Вызовы Codex идут headless с тем же bypass-флагом, что и существующий конвейер репозитория `kb-*-codex.sh` (неинтерактивный запуск его требует). Активная митигация — изоляция на уровне промпта выше. Дальнейшее ужесточение песочницы Codex — отдельное осознанное решение, не требуется для консервативного режима по умолчанию, который Codex вообще не зовёт.

### Граница данных

Deep-проверка (`--deep`) отправляет текст утверждения и URL источников в Codex, который делает web-поиск — поэтому локально полученный текст достигает внешних поисковых сервисов **только при `--deep`**. Консервативный режим по умолчанию этого не делает.

### Локальный скан секретов

`npm run release-check` (и `npm run public-check`) сканируют не только то, что может уйти в публику, но и локальный кэш извлечения (`.mnemazine/cache/extracted/`) на токен-подобные секреты (API-ключи, токены, приватные ключи), потому что захваченные скриншоты или PDF могут содержать креды, которые иначе утекли бы в синтезированные заметки. Захваченный секрет роняет гейт.
